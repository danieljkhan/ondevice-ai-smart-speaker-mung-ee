# ADR 0058: Wave 2 T2.1 TTS Resident — Deferred to Wave 3 due to LLM memory budget regression

- **Status**: Accepted
- **Date**: 2026-04-15

## Context

Wave 2 plan v2 (Plan Gate v2 APPROVED) targets first_sound ≤ 8.0 s via four
tracks: T2.0 (reporting/instrumentation), T2.1 (TTS resident), T2.2 (TTS
sentence-level chunking), T2.3 (LLM prefix caching), and Option B
(max_tokens 60 env override). Implementation began in order. T2.0 landed
cleanly (Codex 902 s, +12 tests, +728/-5 lines). T2.1 also landed cleanly
on the implementation side (Codex 834 s, +9 tests, +323/-10 lines). The
TTS-resident mechanism itself works exactly as designed — the
`avg_tts_load_ms` metric drops from 940.8 ms (Wave 1 v2 baseline) to 52.4
ms in the tmux re-measurement on 2026-04-15. The T2.1 acceptance gate
`avg_tts_load_ms ≤ 100` passes by a wide margin.

The problem is the second-order effect on the LLM stage. With TTS
resident across turns, peak resident memory before LLM load increases by
roughly 450 MB. On Jetson Orin Nano 8 GB unified memory the LLM still
fits, but `MemFree` at LLM-load time drops from the Wave 1 v2 ~6,800 MB
(after passwordless sudo cache drop, ADR 0057) to under 600 MB. With
`MemFree < partial_threshold (3,000 MB)` the manager's
`_select_gpu_layers()` falls back to `n_gpu_layers=10` (safe partial
GPU offload) instead of `-1` (full offload). LLM TTFT and generation both
roughly double in this state.

The `nohup`-based first measurement showed `avg_first_sound_ms = 12,796`
(+1,793 ms regression vs baseline 11,003.8). This was partly inflated by
CUDA zombie accumulation from the off-tmux runner (CLAUDE.md §2 violation;
the original 2026-04-03 evidence already documented 3 GB+ leaks under
`nohup &` SSH disconnects). The tmux re-measurement after explicit zombie
kill + cache drop showed `avg_first_sound_ms = 11,292` (+288 ms vs
baseline). The remaining +288 ms is real, not a measurement artifact: TTS
resident takes 450 MB even when the test environment is clean, and that
margin is what pushes `MemFree` below the partial-offload threshold.
Net effect for T2.1 alone is therefore slightly negative (-888 ms TTS
load saved, +1,176 ms LLM regression added, net +288 ms slower) on the
current Jetson Orin Nano 8 GB unified-memory configuration with the
default `_select_gpu_layers()` policy.

## Decision

Wave 2 T2.1 TTS resident is **deferred to Wave 3**. The implementation
itself (`tts_resident` flag, `unload_tts(force=True)`,
`guard_tts_resident_memory()`, pipeline call-site branching) is kept in
the codebase so the option remains available behind the
`MUNGI_TTS_RESIDENT` environment variable, but the **default stays
`False`** for production runs and the **acceptance gate dependency on
T2.1 is removed from the Wave 2 first_sound budget**. Wave 2
implementation continues with T2.2 (sentence-level chunking) and T2.3
(LLM prefix caching), which do not add resident memory pressure on the
LLM load path. Wave 3 will revisit T2.1 only after one of the following
is in place: (a) LLM-aware load ordering that guarantees full GPU
offload before TTS becomes resident, (b) a dynamic
`_select_gpu_layers()` policy that holds the LLM at full offload even
when `MemFree` is briefly low (e.g. by reserving GPU memory at
`Llama.__init__` time), or (c) hardware migration to Orin NX 16 GB
unified memory which removes the budget contention entirely.

## Consequences

Positive: (a) Wave 2 timeline is preserved — implementation continues
immediately with T2.2 and T2.3 instead of stalling on a memory-budget
redesign that does not directly contribute to Wave 2's hard latency
gates; (b) the T2.1 mechanism is documented and reachable behind a flag,
so future Wave 3 work does not have to re-implement TTS resident from
scratch; (c) the operating principle ("TTS resident saves 890 ms on TTS
load but costs 1,176 ms on LLM regression on this hardware") is now
recorded in the worklog and this ADR, preventing the team from
re-attempting T2.1 without addressing the LLM ordering. Trade-offs: (a)
Wave 2 first_sound budget loses 890 ms of theoretical headroom — Wave 2
must rely on T2.2 (-621 ms) + T2.3 (-547 ms) + Option B (-300 ms) to
reach the 8.0 s target, total -1,468 ms vs the 3,000 ms gap, which means
Wave 2 will likely land at first_sound ≈ 9.5 s rather than ≤ 8.0 s
without a stretch option (e.g. STT GPU build); (b) the TTS-resident code
path remains exercised by tests and `MUNGI_TTS_RESIDENT=1` runs, so
maintenance cost is non-zero even though the feature is not enabled by
default; (c) operators who explicitly opt in via the env var must
understand they will see the LLM regression unless they also accept
load-order workarounds. Operational impact: Wave 2 plan v2 §8
acceptance gates are revised — `avg_tts_load_ms ≤ 100` is dropped from
the Wave 2 hard list (it can still be measured for reference when T2.1
is opted in), and the first_sound ≤ 8.0 s target is acknowledged as
likely needing Wave 3 follow-up.

The CLAUDE.md §2 E2E Test Execution Workflow (tmux required) is
underscored by this incident: the `nohup`-based first measurement
overstated the regression by ~1,500 ms because of CUDA zombie
accumulation. Operators MUST run all multi-turn E2E tests inside a tmux
session, and the runner's `--skip-preflight` flag MUST NOT be combined
with off-tmux execution. PM enforced the rule in the second measurement
(zombie kill + page-cache drop + tmux session) and the regression
shrank from +1,793 ms to +288 ms, attributing the ~85% gap to nohup
cleanup hygiene rather than to T2.1 itself.

## Update — 2026-04-15 오후 (T2.2 overhead fix measurement)

The original T2.1 defer rationale was a +288 ms regression from LLM
memory contention when TTS resident stays across turns. Subsequent
T2.2 measurement on 2026-04-15 uncovered a separate latent bug: the
pipeline was redundantly loading/unloading the Supertonic engine on
every streaming turn even when `_run_tts` had been patched to a
no-op (the `_maybe_externalize_tts` context manager only intercepted
the synthesis hook, not the ModelManager lifecycle calls at
`core/pipeline.py:700-713, 815-825`).

The B-fix (`scripts/e2e_qwen3_asr_mix.py` — extended
`_maybe_externalize_tts` to also patch `_maybe_unload_tts_after_
success`, `_mm.load(ModelType.TTS)`, and `_mm.unload_tts`) eliminates
~1 s of per-turn wasted load/unload around the no-op synthesis hook.

The 2026-04-15 Jetson tmux measurement compared three runs:

| Config | env | avg_first_sound_ms | avg_tts_load_ms |
|---|---|---:|---:|
| T2.0 baseline (ref) | LLM_RESIDENT+STT_RESIDENT | 11,292 | 925 |
| B (pipeline fix, no resident) | + TTS_STREAMING | 12,042 | 0.009 |
| C (B + resident) | + TTS_RESIDENT + TTS_STREAMING | 12,001 | 0.010 |

C vs B delta = -41 ms (within measurement noise). The `avg_tts_load_
ms` stays at ~0.01 ms in both B and C because the pipeline no longer
invokes the loader at all — the streaming helper's internal
`_STREAMING_ENGINE_CACHE` owns the Supertonic engine across turns
without needing ModelManager-level residence.

### Consequence for T2.1

The B-fix invalidates the original defer condition:
1. The pipeline never asks ModelManager to load TTS during streaming
   turns, so TTS residence has no effect on pipeline-driven memory
   pressure. The LLM memory-contention path that produced the +288 ms
   regression no longer applies when B is active.
2. T2.1 becomes **redundant** under B (not beneficial, not harmful) —
   because streaming's own cache already keeps one engine warm.
3. Outside streaming mode, T2.1's original measurement still applies:
   the +288 ms LLM regression reappears if TTS is held resident while
   the pipeline owns the TTS lifecycle.

### Revised Wave 3 direction

Wave 3 revisit of T2.1 must treat the B-fix as a prerequisite, and
any future measurement of T2.1 value should be done under
non-streaming workloads (where TTS-resident actually skips reloads)
rather than streaming workloads (where B already externalizes
lifecycle).

### Measurement reporting gap (follow-up)

The reported `avg_first_sound_ms` in streaming mode still uses the
full TTS synthesis duration instead of the first-chunk dispatch
time. Perceived first-sound for children using streaming mode is
approximately:

```
perceived_first_sound
  = VAD + STT + LLM_load + LLM_inference + TTS_first_chunk
  = 373 + 5828 + 160 + 3608 + 978
  ≈ 10,947 ms   (C run, within noise of 11,004 ms Wave 1 baseline)
```

A follow-up task will update the summary reporting to switch the
first-sound formula to `tts_first_chunk_ms` when streaming is active,
so the metric reflects the real user-perceived latency rather than
the full-synthesis bound.

## Update 2 — 2026-04-16

Wave 3 Codex review round 2 finding B-5 confirmed that TTS resident retry
is not a useful latency lever under the streaming B-fix. In the Wave 2 final
streaming measurement, `avg_tts_load_ms = 0.009`, which leaves effectively
nothing for ModelManager-level TTS residence to save.

Wave 3 task T3.2 is therefore dropped. The T2.1 TTS resident retry deferral
is permanent for this product generation unless the product later returns to
a non-streaming TTS lifecycle where ModelManager owns the engine reload cost.
