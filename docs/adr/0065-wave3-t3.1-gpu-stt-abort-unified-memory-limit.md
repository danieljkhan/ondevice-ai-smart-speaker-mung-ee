# ADR 0065: Wave 3 T3.1 GPU STT abort — Orin Nano Super unified-memory limit

- **Status**: Accepted
- **Date**: 2026-04-18

## Context

Wave 3 Plan `docs/archived/dev-plan/2026-04-16-Wave3-Plan.md` §3 T3.1 targeted `avg_stt_total_ms` 6,171 ms → 2,500–3,000 ms (−3,100 to −3,600 ms on `first_sound`) by rebuilding `sherpa-onnx 1.12.38` with CUDA 12.6 + ONNX Runtime 1.18.1 GPU on Jetson Orin Nano Super (sm_87). T3.1 was the **primary latency lever** of Wave 3 — without it the stack cannot reach the 8,000 ms first_sound target.

The infrastructure landed across five PRs on 2026-04-18:

GitHub Release `v1.12.38-sherpa-gpu` was uploaded with the built wheel (93.7 MB, SHA256 `858f730cc605c6289a17fbde22d399e38c239e9e5636d1b8df8825b606857bde`).

The Jetson deployment + CUDA smoke test (`OfflineRecognizer.from_qwen3_asr(provider="cuda")`) confirmed that the wheel is genuinely CUDA-enabled (no `Please compile with -DSHERPA_ONNX_ENABLE_GPU=ON. Fallback to cpu!` warning). Plan §3 T3.1 gates (a)–(c) all cleared; the hardware-execution question remained.

Wave 3 Plan §5 M1 measurement was executed on 2026-04-18 22:49–22:58 KST against the same `qwen3_mix` 24-round bilingual benchmark as Wave 2 final, with the sole delta `MUNGI_STT_PROVIDER=cuda`. Run output: `/var/lib/mungi/conversations/qwen3_mix_20260418_224950/`.

## Decision

**T3.1 is aborted.** The `MUNGI_STT_PROVIDER` env default remains `cpu`; no follow-up PR will flip it. The Orin Nano Super platform does not admit a net-positive GPU STT stack with the current LLM resident configuration. This is platform-architectural, not a software bug — no further software investment on GPU STT is scheduled for Wave 4.

Overall: `avg_first_sound_ms` is **+2,307 ms (21%) worse** than Wave 2 baseline, not better. First-order Wave 3 rationale inverted.

## Root cause — Orin Nano Super unified-memory contention

**Hardware architecture**: Orin Nano Super uses **unified memory** — GPU VRAM and system RAM are the same physical pool (8 GB total). When sherpa-onnx runs with `provider="cuda"`, its CUDA allocations come from the same pool as the resident LLM's GPU-offloaded weights + KV cache + TTS engine + system baseline.

**Measured behavior**:
1. **STT inference latency unchanged**: 5,833 → 5,811 ms (−0.4%, noise level). sherpa-onnx CUDA provider does not accelerate Qwen3-ASR int8 on Orin Nano Super compared to the CPU provider. Possible reasons: (a) int8 quantization already optimized for CPU SIMD; (b) Orin Nano Super's GPU compute ceiling is saturated at the model size; (c) unified-memory transfer overhead cancels theoretical GPU speedup. Exact root cause not isolated — moot given the other cost below.
2. **STT process RAM footprint**: 1,668 MB (measured via `free -m` delta during smoke test; see `docs/runbooks/weekly/archive/2026-04-18-daily-worklog.md` §15). Plan §4 arena estimate was 300 MB; actual includes model + CUDA context + ORT GPU libs + arena → **5.5× over estimate**.
3. **LLM slowdown**: TTFT +547 ms (+42%), total +1,772 ms (+50%). Root cause is memory contention: LLM `n_gpu_layers=-1` (full offload) contends with sherpa-onnx CUDA arena on the same unified pool → either layer offload silently degrades or the memory-guard (ADR 0060) demotes to partial offload. Both outcomes produce the observed regression.
4. **Memory pressure constant**: 23 WARNING memory events across 24 rounds (5,800–5,900 MB usage, vs CRITICAL threshold 6,500 MB). Stability holds (0 CRITICAL, 0 force-unload), but the slack is gone.

The net effect is a **negative-sum trade**: the GPU STT arena takes resources from the LLM GPU offload, and the STT itself doesn't benefit from the move.

## Plan §6 risk triggers

- **R3.1.2** ("GPU not 2–3x faster on Orin Nano's limited VRAM/compute. Measure first; abort stack if <1.5x.") — measured speedup is effectively 1.0x. **Triggered.**
- **R3.1.3** ("LLM + STT GPU + TTS + LlamaCache collide under memory pressure. Mitigation: ADR 0060 Fix-D + VAD protection already deployed; T3.5 removes LlamaCache pressure entirely.") — even with LlamaCache removed (T3.5) and ADR 0060 Fix-D active, LLM latency regressed. **Triggered.**

Both risks fired as predicted by the Wave 3 Plan design review.

## What is removed as a consequence

- `.codex/pending-task-stt-provider-flip.md` — the staged Codex task for flipping `stt_device` default from `cpu` to `cuda`. Deleted. Wave 3 Plan §3 T3.1 (c) is now formally retired; the env reader landed in PR #30 is the terminal deliverable on that track.
- Wave 3 M1 stacked measurement sequence (M2/M3 per Plan §5) — dependent on T3.1 as the memory baseline shift. Now meaningless without a GPU-STT default flip.

## Wave 3 net state after abort

Landed deltas (vs Wave 2 baseline):
- **T3.0 stability telemetry**: 0 ms direct, enabler (merged PR #18).
- **T3.4 history window / conversation-per-lang**: 0 ms standalone, enabler for T3.3 (merged PR #20).
- **T3.5 LlamaCache removal**: 0 ms (debt; default already OFF at Wave 2 measurement time) (merged PR #19).
- **T3.1 GPU STT**: **REJECTED** (measurement, this ADR).
- **T3.3 low-level LLM chat**: NOT YET MEASURED. Micro-spike (d0) still pending per Plan §3.

`avg_first_sound_ms` landed-state estimate: **10,912 ms** (Wave 2 baseline held; no regression from the three landed tracks).

Gap to 8,000 ms target: **2,912 ms** (same as before Wave 3 began). Only T3.3 (−524 ms predicted) remains as a potential path. Without T3.1 the 8,000 ms target is not attainable in this Wave.

## Wave 3 closure path

1. **T3.3 micro-spike** (`scripts/_spike_llama_lowlevel.py`, Jetson-only, not tracked) — Plan §3 T3.3 (d0). Validate llama-cpp-python 0.3.17 sampler-chain API + byte-parity. Next session work.
2. **On spike PASS**: T3.3 (a)–(f) Codex delegation → measurement → Wave 3 close with T3.3 +T3.0+T3.4+T3.5 delta (~−500 ms net; `first_sound ≈ 10,400 ms`; 8,000 target unmet; document remaining gap as Wave 4 software scope).
3. **On spike FAIL**: Wave 3 close with T3.0+T3.4+T3.5 only (0 ms net; `first_sound ≈ 10,912 ms`); scope Wave 4 separately.

In both cases Wave 3 will formally miss its 8,000 ms target. This ADR acknowledges that outcome up-front and decouples the T3.1 decision from the Wave 3 closing narrative.

## Hardware scope (unchanged)

Per user directive 2026-04-16 and reaffirmed 2026-04-18: **no hardware migration** (Orin NX 16 GB, Orin AGX, or other) is in scope for Wave 3 or Wave 4. Mungi continues to target Orin Nano Super 8 GB. GPU STT on this platform is a **permanent-software decision** per this ADR, not a "wait for hardware upgrade" deferral.

## Consequences

**Positive**:
- Decision recorded with measurement evidence; future readers understand why the `MUNGI_STT_PROVIDER=cuda` path exists but is not default.
- Plan §6 R3.1.2 / R3.1.3 risk register now has historical validation — signal for future waves when considering GPU offload on unified-memory Jetsons.
- Build infra (PR #24/#27/#28/#29) remains reusable for verification or research — not a scope-expansion loss.
- Env reader (PR #30) preserves opt-in flexibility without production cost.

**Negative**:
- Wave 3 8,000 ms target unachievable.
- ~6 hours of session work (4-5 PRs, 3 build iterations, M1 run) produced a NET-ZERO delta to production latency on this axis.
- Narrative for Wave 3 close weaker than hoped; T3.3 carries sole remaining upside.

**Neutral**:
- `docs/archived/dev-plan/2026-04-16-Wave3-Plan.md` §3 T3.1 is now closed with a documented outcome. No plan-document edit required; this ADR is the authoritative record.
