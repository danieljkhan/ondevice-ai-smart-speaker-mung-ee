# ADR 0061: Wave 2 T2.3 Phase 2 — `create_chat_completion` ignores pre-loaded KV state; deferred to Wave 3

- **Status**: Accepted
- **Date**: 2026-04-15

## Context

Wave 2 T2.3 targeted −547 ms on `avg_llm_ttft_ms_after_first` by reusing
the system-prompt KV prefix across turns. The original plan assumed
`Llama.create_chat_completion(cache_prompt=True)` would be available in
llama-cpp-python 0.3.17; that kwarg exists only in the llama.cpp HTTP
server API, not in the Python bindings (verified 2026-04-15,
`inspect.signature(Llama.create_chat_completion)` returns 28 params, no
`cache_prompt`).

Two substitute implementations were attempted:

### Attempt 1 — `LlamaCache` attach (commits `9905b97`, `12bb1cb`)

- Attach `LlamaRAMCache(capacity_bytes=200 MB)` via `llm.set_cache(...)`.
- `create_chat_completion` internally performs a longest-token-prefix
  match against the cache's stored `LlamaState` snapshots.
- 2026-04-15 Jetson 24-round measurement: hit rate **0.4 %**.
- Root cause: `core/pipeline.py::clear_history()` runs every turn,
  so the messages list reorders per turn. Cache keys are full-prompt
  token hashes; the common prefix is short enough that the LRU
  fetcher rarely returns a useful snapshot.

### Attempt 2 — explicit `save_state` / `load_state` snapshot (commit `789e8aa`)

- `prepare_system_state_snapshot(llm, prompt)` runs `reset() → eval(tokens)
  → save_state()` once at LLM load, per-language (KO + EN).
- Before each `create_chat_completion`, the pipeline calls
  `restore_system_state_snapshot(llm, state)` which invokes
  `llm.load_state(state)`.

### Direct API feasibility test (2026-04-15 21:43 KST, Jetson)

```python
llm = Llama(model_path=..., n_gpu_layers=-1, n_ctx=2048, flash_attn=True)
system_prompt = "You are a helpful assistant. " * 50   # 301 tokens
tokens = llm.tokenize(system_prompt.encode("utf-8"), add_bos=True, special=True)

llm.reset()
llm.eval(tokens)
# After eval, n_tokens = 301 ✓
state = llm.save_state()

llm.reset()
# After reset, n_tokens = 0
llm.load_state(state)
# After load_state, n_tokens = 301 ✓ (context was restored)

messages = [{"role": "system", "content": system_prompt},
            {"role": "user",   "content": "Hi"}]
t0 = time.monotonic(); llm.create_chat_completion(messages=messages, max_tokens=3)
ttft_restored = time.monotonic() - t0   # 559 ms, n_tokens end = 320

llm.reset()
t0 = time.monotonic(); llm.create_chat_completion(messages=messages, max_tokens=3)
ttft_fresh = time.monotonic() - t0      # 459 ms, n_tokens end = 320

# Speedup = ttft_fresh / ttft_restored = 0.82x
```

The restore + `create_chat_completion` path is **slower** than starting
from a fresh context by ~100 ms (the cost of `load_state` itself). The
final `n_tokens` is identical in both paths, which means
`create_chat_completion` discards the pre-loaded context and
re-tokenizes the full messages list regardless.

### Jetson 24-round measurement with Phase 2 enabled

| Metric | Phase 1 (q8_0) | Phase 2 (snapshot ON) | Delta |
|---|---:|---:|---:|
| CRITICAL memory events | 1 | 0 | −1 |
| `avg_llm_ttft_ms_after_first` | 2,714 | 2,190 | −524 |
| `avg_llm_ms` | 6,441 | 4,506 | −1,935 |
| `avg_first_sound_ms` | 14,576 | 13,856 | −720 |

Observed Jetson deltas look like a partial improvement, but the direct
API test proves the speedup is attributable to **reduced memory
pressure** (CRITICAL events dropped from 1 to 0, reducing LLM full
reloads) rather than to prefix reuse. The snapshots capture but the
restore is effectively a no-op on `create_chat_completion`'s
generation path.

## Decision

T2.3 Phase 2 is deferred to Wave 3. The code is retained (commit
`789e8aa`) but **opt-in only**; default
`PipelineConfig.llm_system_state_snapshot = False` keeps production
runs on the fresh-context path.

The feature cannot meet its design goals (`avg_llm_ttft_ms_after_first
< 1,200`, `avg_first_sound_ms ≤ 11,500`) with the current
`create_chat_completion`-centric API surface. Three Wave 3 retry paths
exist:

1. **Low-level `Llama.__call__()` pipeline** — bypass
   `create_chat_completion`, tokenize manually, call `llm()` with the
   incremental user tokens only after `load_state`. This requires
   reimplementing the chat-template rendering inline.
2. **Dedicated HTTP server mode** — run `llama_cpp.server` with
   `cache_prompt=True` and switch the pipeline to HTTP calls. Adds a
   deployment dependency but gains the documented prefix cache.
3. **Wait on upstream** — future llama-cpp-python releases may expose
   `cache_prompt` directly. Track the upstream repo.

## Consequences

### Positive

- Phase 2 code is shipped behind an opt-in flag so the investigation
  is reproducible without affecting production.
- The direct API test is captured in this ADR; future contributors
  can re-run it in 3 lines when evaluating new bindings.
- The Phase 1 quantization (`type_k='q8_0'`) shipped in commit
  `84c6b42` remains useful as a standalone memory-headroom lever
  (≈−200 MB of KV footprint) and is **not** affected by this ADR.
- ADR 0060 (VAD protection, layered eviction) and the memory-guard
  hardening from today's session are orthogonal to T2.3 and continue
  to provide their designed benefits.

### Negative / Risks

- The `avg_llm_ttft_ms_after_first` regression (~2,700 ms vs Wave 1
  baseline of ~1,400 ms) remains unaddressed within Wave 2.
- `avg_first_sound_ms ≈ 13,800 ms` vs the Wave 2 target of ≤ 8,000 ms
  represents a ~5,800 ms gap. Wave 2 will close without hitting its
  first_sound goal; Option B (`MUNGI_LLM_MAX_TOKENS=60`) will close
  ~300 ms of that gap but the structural gap requires Wave 3.
- The `LlamaCache` attach path is still on by default. After this
  ADR lands, a follow-up commit should flip
  `PipelineConfig.llm_cache_prompt` default to `False` too, since
  its benefit is negligible and its memory pressure is real (ADR
  0058 Update § for the memory implications).

## Update — 2026-04-16 — LlamaCache path removed (Wave 3 T3.5)

Wave 3 T3.5 removed the Attempt 1 `LlamaCache` attach code path entirely.
The removed surface includes the opt-in prefix-cache environment handling,
the external cache attach/detach helpers, and the public helper that cleared
that external cache.

The separate llama-cpp-python `cache_prompt` argument remains available in
`run_chat_generation()` for future API-level experiments. Pipeline callers
no longer pass it as part of this cleanup.

Landing commit: `f39240c`.
